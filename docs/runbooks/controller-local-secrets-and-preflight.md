# Controller-Local Secrets And Preflight Runbook

## Purpose

This runbook defines the machine-readable controller-local secret contract and the standard preflight command for long-running automation.

## Manifest

The canonical manifest lives at [config/controller-local-secrets.json](/Users/live/Documents/GITHUB_PROJECTS/proxmox_florin_server/config/controller-local-secrets.json).

Workflow ownership now lives in [config/workflow-catalog.json](/Users/live/Documents/GITHUB_PROJECTS/proxmox_florin_server/config/workflow-catalog.json).

It records:

- local file secrets under `.local/`
- required environment variables
- whether a secret is provided externally or generated by repo automation
- which automation surfaces depend on each secret

## Primary Command

List available workflows:

```bash
make preflight
```

Check one workflow before starting a converge:

```bash
make preflight WORKFLOW=converge-monitoring
```

The same command is invoked automatically by the long-running Make targets that depend on controller-local secret material.

To inspect the full workflow contract, including entry points and runbooks:

```bash
make workflow-info WORKFLOW=converge-monitoring
```

For the PostgreSQL tailnet DNS workflow, use the managed entry point:

```bash
HETZNER_DNS_API_TOKEN=... make database-dns
```

## Typical Workflows

- `make preflight WORKFLOW=harden-security`
- `make preflight WORKFLOW=provision-api-access`
- `make preflight WORKFLOW=converge-monitoring`
- `make preflight WORKFLOW=deploy-uptime-kuma`
- `make preflight WORKFLOW=configure-backup-vm`
- `make preflight WORKFLOW=database-dns`

## Generated Local Material

Some controller-local files are created by repo automation and are not required before the generating workflow starts. Examples:

- `.local/proxmox-api/lv3-automation-primary.json`
- `.local/monitoring/proxmox-writer.token`
- `.local/monitoring/guest-writer.token`
- `.local/proxmox-backup/backup-lv3-token.json`
- `.local/tfa/proxmox-ops-pam-totp.json`

The preflight command reports these as generated outputs for the relevant workflow instead of failing on them up front.

## Blocked Paths

`make preflight WORKFLOW=configure-backups` is expected to fail.

That workflow still points at the superseded ADR 0020 external CIFS secret set and is intentionally marked blocked in the workflow catalog. Use the PBS-based backup path from ADR 0029 instead.

## Notes

- The secret manifest describes secret inventory only. The workflow catalog owns workflow ids and secret usage.
- Keep `.local/` out of git and treat it as part of the real operating contract for the controller machine.
- If a new playbook or script needs local secret material, update the secret manifest and workflow catalog in the same change.
