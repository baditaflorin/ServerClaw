# Controller-Local Secrets And Preflight Runbook

## Purpose

This runbook defines the machine-readable controller-local secret contract and the standard preflight command for long-running automation.

## Manifest

The canonical manifest lives at [config/controller-local-secrets.json](../../config/controller-local-secrets.json).

Workflow ownership now lives in [config/workflow-catalog.json](../../config/workflow-catalog.json).

Fresh-worktree bootstrap requirements now live in [config/worktree-bootstrap-manifests.json](../../config/worktree-bootstrap-manifests.json).

It records:

- local file secrets under `.local/`
- repo-relative `.local/...` paths that remain portable across worktrees and fresh forks
- required environment variables
- whether a secret is provided externally or generated by repo automation
- which automation surfaces depend on each secret
- generated worktree artifacts that must exist before a governed run starts
- controller-local helper paths or caches that should be checked early and reported clearly

## Primary Command

List available workflows:

```bash
make preflight
```

Check one workflow before starting a converge:

```bash
make preflight WORKFLOW=converge-monitoring
```

The same command is invoked automatically by the generic `live-apply-group`, `live-apply-service`, `live-apply-site`, `live-apply-waves`, and `post-merge-gate` entrypoints before they continue into mutation or validation work.

To inspect the full workflow contract, including entry points and runbooks:

```bash
make workflow-info WORKFLOW=converge-monitoring
```

For the PostgreSQL tailnet DNS workflow, use the managed entry point:

```bash
HETZNER_DNS_API_TOKEN=... make database-dns
```

## Typical Workflows

- `make preflight WORKFLOW=harden-security`
- `make preflight WORKFLOW=provision-api-access`
- `make preflight WORKFLOW=converge-monitoring`
- `make preflight WORKFLOW=deploy-uptime-kuma`
- `make preflight WORKFLOW=configure-backup-vm`
- `make preflight WORKFLOW=database-dns`

## Generated Local Material

Some controller-local files are created by repo automation and are not required before the generating workflow starts. Examples:

- `.local/proxmox-api/lv3-automation-primary.json`
- `.local/monitoring/proxmox-writer.token`
- `.local/monitoring/guest-writer.token`
- `.local/proxmox-backup/backup-lv3-token.json`
- `.local/tfa/proxmox-ops-pam-totp.json`

The preflight command reports these as generated outputs for the relevant workflow instead of failing on them up front.

The bootstrap manifest also allows generated worktree artifacts to be materialized automatically before the main workflow continues. The current shared example is the edge-publication portal set:

- `build/ops-portal/`
- `build/changelog-portal/`
- `build/docs-portal/`

When a governed workflow declares those artifacts, `make preflight` repairs them automatically from the current worktree instead of letting the apply fail later inside `nginx_edge_publication`.

To exercise that path directly:

```bash
make preflight WORKFLOW=configure-edge-publication
```

## Blocked Paths

`make preflight WORKFLOW=configure-backups` is expected to fail.

That workflow still points at the superseded ADR 0020 external CIFS secret set and is intentionally marked blocked in the workflow catalog. Use the PBS-based backup path from ADR 0029 instead.

## Notes

- The secret manifest describes secret inventory only. The workflow catalog owns workflow ids and secret usage.
- The bootstrap manifest owns worktree-generated artifacts and non-secret local helper-path checks for governed automation.
- Keep `.local/` out of git and treat it as part of the real operating contract for the controller machine.
- Use [config/examples/reference-controller-local-secrets.json](../../config/examples/reference-controller-local-secrets.json) as the public-safe starter contract for a new fork, then replace only the entries your environment actually needs.
- If a new playbook or script needs local secret material, update the secret manifest and workflow catalog in the same change.
- If a fresh worktree depends on generated local output or a non-secret controller path, update the bootstrap manifest catalog in the same change.
