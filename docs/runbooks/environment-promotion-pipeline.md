# Environment Promotion Pipeline

## Purpose

This runbook documents ADR 0073: the governed promotion path from a validated staging receipt to a production live apply.

It covers:

- repository validation before a production promotion
- staging receipts under `receipts/live-applies/staging/`
- promotion gate enforcement through `scripts/promotion_pipeline.py` backed by the shared ADR 0230 OPA policy bundle
- production promotion execution through `make promote`
- linked promotion receipts under `receipts/promotions/`
- break-glass bypass audit emission when `bypass_promotion=true` is used on a direct production live apply

## Preconditions

Before running the promotion workflow, confirm:

1. the change is on a `codex/` or `operator/` branch, not `main`
2. a recent staging receipt exists under `receipts/live-applies/staging/`
3. the staging receipt reports only `pass` verification results for the service being promoted
4. no open `critical` ADR 0071 finding exists for the service in `.local/platform-observation/latest/findings.json`
5. at least one human operator approval is recorded for the `promote-to-production` command contract
6. the shared policy bundle validates cleanly: `python3 scripts/policy_checks.py --validate`

## Entrypoints

- inspect the workflow contract: `make workflow-info WORKFLOW=deploy-and-promote`
- inspect the command contract: `make command-info COMMAND=promote-to-production`
- dry-run the gate and approval path: `python3 scripts/promotion_pipeline.py --check-gate --service <service-id> --staging-receipt receipts/live-applies/staging/<receipt>.json`
- dry-run the full promotion: `make promote SERVICE=<service-id> STAGING_RECEIPT=receipts/live-applies/staging/<receipt>.json DRY_RUN=true`
- run the full promotion: `make promote SERVICE=<service-id> STAGING_RECEIPT=receipts/live-applies/staging/<receipt>.json`

## Delivered Surfaces

The promotion workflow manages these repo surfaces:

- `scripts/promotion_pipeline.py`
- `policy/decisions/release_promotion.rego`
- `scripts/policy_checks.py`
- `receipts/live-applies/staging/`
- `receipts/promotions/`
- `docs/schema/promotion-receipt.json`
- `config/command-catalog.json` command `promote-to-production`
- `config/workflow-catalog.json` workflow `deploy-and-promote`
- seeded Windmill wrapper `f/lv3/deploy_and_promote`

## Promotion Flow

1. `make validate` runs on the branch being promoted.
2. The promotion gate validates the staged receipt path, freshness, verification outcomes, command-catalog approval, capacity headroom, standby policy, and SLO state through one shared OPA decision input.
3. If the gate passes, `make live-apply-service service=<service-id> env=production` runs through the existing service playbook path.
4. A production live-apply receipt is written under `receipts/live-applies/`.
5. A promotion receipt is written under `receipts/promotions/` linking the staging evidence to the production mutation.

## Verification

Run these checks after implementing or changing the promotion flow:

1. `python3 scripts/promotion_pipeline.py --validate`
2. `python3 scripts/policy_checks.py --validate`
3. `python3 -m unittest discover -s tests -p 'test_promotion_pipeline.py'`
4. `make command-info COMMAND=promote-to-production`
5. `make workflow-info WORKFLOW=deploy-and-promote`
6. `make promote SERVICE=grafana STAGING_RECEIPT=receipts/live-applies/staging/example.json DRY_RUN=true`

## Break-Glass

If production must be mutated without a staging promotion:

1. run the direct live apply against production with `bypass_promotion=true` in `EXTRA_ARGS`
2. ensure the direct apply path emits the `promotion_bypassed` audit event
3. record the incident note and recovery context within 24 hours

Examples:

```bash
make live-apply-service service=grafana env=production EXTRA_ARGS='-e bypass_promotion=true'
make live-apply-group group=automation env=production EXTRA_ARGS='-e bypass_promotion=true'
```

## Notes

- The first implementation keeps staging receipts and promotion receipts fully repo-managed even though the actual stage/prod apply commands still flow through the existing Ansible playbook entrypoints.
- The seeded Windmill wrapper calls the same `scripts/promotion_pipeline.py` entrypoint used by operators on the controller, and ADR 0230 now routes the release decision through the shared `policy/decisions/release_promotion.rego` bundle instead of duplicating that logic in Python or Windmill-only code.
- The current production receipt generated by `make promote` is intentionally concise and links back to the staging receipt plus the promotion receipt for the full chain of evidence.
