# Proxmox Guest Operations via QAPI (Zero-SSH)

## Overview

All controller-initiated imperative operations targeting VM guests use the
Proxmox QAPI guest-exec path (`ProxmoxClient.guest_exec`) rather than direct
SSH. This is the pattern mandated by [ADR 0342](../adr/0342-zero-ssh-guest-operations-via-proxmox-qapi.md).

**Why not SSH?**

- SSH requires per-environment key distribution and per-VM network reachability
  on port 22.
- The Proxmox API is already reachable via the Tailscale-accessible endpoint
  (`https://100.64.0.1:8006/api2/json`) and a single PVE API token covers all
  VMs on the node.
- `guest_exec` returns structured `(returncode, stdout, stderr)` enabling
  proper error handling, retry logic, and test doubles in CI.
- The QEMU guest agent (QGA) is pre-installed on all repo-managed VMs via the
  `linux_guest_runtime` role, so no additional setup is needed per target VM.

## Prerequisites

| Item | Location |
|---|---|
| Proxmox API auth file | `.local/proxmox-api/lv3-automation-primary.json` |
| Topology snapshot | `scripts/topology-snapshot.json` (generated by `generate_topology_snapshot.py`) |
| `proxmox_tool.py` | `scripts/proxmox_tool.py` |
| `coolify_tool.py` | `scripts/coolify_tool.py` (Coolify-specific operations only) |

The auth file format:

```json
{
  "api_url": "https://100.64.0.1:8006/api2/json",
  "authorization_header": "PVEAPIToken=lv3-automation@pve!primary=<secret>"
}
```

## Common Operations

### List Docker containers on a VM (`docker-ps`)

```bash
python3 scripts/proxmox_tool.py \
  --auth-file .local/proxmox-api/lv3-automation-primary.json \
  --env prod docker-ps --vmid 171
```

Replace `171` with the target VMID. VMID assignments are in
`versions/stack.yaml` under `desired_state.guest_provisioning.guest_vmids`.

### Run an arbitrary command inside a VM (`guest-exec`)

```bash
python3 scripts/proxmox_tool.py \
  --auth-file .local/proxmox-api/lv3-automation-primary.json \
  --env prod guest-exec --vmid 170 \
  --command '["systemctl", "status", "docker"]'
```

The `--command` argument accepts a JSON array of strings. The command runs
inside the VM via the QEMU guest agent; there is no SSH session.

### Install an SSH public key into a VM's authorized_keys (`install-key`)

Installing SSH keys is the one operation class that inherently targets the SSH
surface. `proxmox_tool.py` provides a `install-key` command that still uses
QAPI guest-exec under the hood to write the key — no SSH is opened:

```bash
python3 scripts/proxmox_tool.py \
  --auth-file .local/proxmox-api/lv3-automation-primary.json \
  --env prod install-key --vmid 171 \
  --public-key "ssh-ed25519 AAAA... operator@controller"
```

### Coolify-specific guest operations

Coolify operations (database exec, cache clear, app migration, deploy key
installation) live in `scripts/coolify_tool.py`, **not** `proxmox_tool.py`.
This is per [ADR 0345](../adr/0345-layered-operator-tool-separation.md).

```bash
# Run a psql query inside the Coolify database container
python3 scripts/coolify_tool.py \
  --auth-file .local/proxmox-api/lv3-automation-primary.json \
  --env prod coolify-db-exec --vmid 171 \
  --query "SELECT id, name FROM teams LIMIT 5;"

# Clear Coolify application cache
python3 scripts/coolify_tool.py \
  --auth-file .local/proxmox-api/lv3-automation-primary.json \
  --env prod coolify-clear-cache --vmid 171

# Run pending database migrations
python3 scripts/coolify_tool.py \
  --auth-file .local/proxmox-api/lv3-automation-primary.json \
  --env prod coolify-migrate-apps --vmid 171

# Install an SSH deploy key into Coolify's known_hosts / agent
python3 scripts/coolify_tool.py \
  --auth-file .local/proxmox-api/lv3-automation-primary.json \
  --env prod coolify-install-deploy-key --vmid 171 \
  --key-file .local/ssh/coolify-deploy-key.pub
```

## Troubleshooting

### QEMU guest agent not running

**Symptom:** `RuntimeError: QAPI exec failed: 500 ... QEMU guest agent is not running`

**Cause:** The `qemu-guest-agent` service inside the VM is stopped or not installed.

**Fix:**

```bash
# Connect via Proxmox console (no SSH required)
# On the target VM:
systemctl start qemu-guest-agent
systemctl enable qemu-guest-agent
```

Alternatively, use the Proxmox web UI: Datacenter → node → VM → Console.

All repo-managed VMs have `qemu-guest-agent` installed via the
`linux_guest_runtime` Ansible role. If it is missing, re-run the role:

```bash
ansible-playbook playbooks/linux_guest_runtime.yml --limit <vmid-or-hostname>
```

### Auth file not found or token invalid

**Symptom:** `FileNotFoundError` or `RuntimeError: HTTP 401 Unauthorized`

**Checks:**

1. Confirm the auth file exists: `ls -l .local/proxmox-api/lv3-automation-primary.json`
2. Verify the token is still valid:
   ```bash
   python3 - <<'PY'
   import json, pathlib, urllib.request, ssl
   p = json.loads(pathlib.Path(".local/proxmox-api/lv3-automation-primary.json").read_text())
   ctx = ssl.create_default_context(); ctx.check_hostname = False; ctx.verify_mode = ssl.CERT_NONE
   req = urllib.request.Request(p["api_url"] + "/version", headers={"Authorization": p["authorization_header"]})
   print(urllib.request.urlopen(req, context=ctx).read().decode())
   PY
   ```
3. If the token is expired or revoked, re-create it via the Proxmox API automation
   runbook: `docs/runbooks/proxmox-api-automation.md`.

### Wrong node name in topology snapshot

**Symptom:** `RuntimeError: node 'pve' not found` or similar.

**Cause:** The node name in `scripts/topology-snapshot.json` does not match the
actual Proxmox node name.

**Fix:** Regenerate the snapshot:

```bash
python3 scripts/generate_topology_snapshot.py --write
```

Then verify the `node` field matches the output of `pvesh get /nodes` on the
Proxmox host.

### Command times out inside the VM

**Symptom:** `RuntimeError: guest_exec timed out after 60s`

**Cause:** The command running inside the VM is taking longer than the default
60-second timeout, or it is hanging waiting for interactive input.

**Fix:**

- Add the `-y` / `--yes` / `--no-input` flags to the inner command to avoid
  interactive prompts.
- Pass `--timeout <seconds>` to `proxmox_tool.py` for longer-running operations.
- Avoid commands with pagers (e.g. pass `--no-pager` to `journalctl`, use
  `| cat` for docker commands).
