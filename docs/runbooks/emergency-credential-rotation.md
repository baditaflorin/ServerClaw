# Emergency Credential Rotation Runbook

**ADR:** 0405
**Trigger:** Public repository exposure (2026-04-11)
**Script:** `scripts/emergency_credential_rotation.py`

---

## When to Use This Runbook

Use this procedure when:
- The codebase or secret structure has been exposed publicly
- A credential compromise is suspected across multiple services
- A full platform rotation is required as part of incident response

For single-secret rotation, use the standard rotation workflow
(`scripts/secret_rotation.py`) documented in ADR 0065.

---

## Prerequisites

1. SSH access to all managed hosts via the bootstrap key
2. Access to the Proxmox host for API token regeneration
3. Access to the Keycloak admin console for client secret regeneration
4. Rename `.local/open-webui/provider.env` and `.local/serverclaw/provider.env`
   to `.bak` (preflight scanner workaround per CLAUDE.md)

---

## Procedure

### Step 0: Assess the Exposure

```bash
# Review what will be rotated
python scripts/emergency_credential_rotation.py --list
```

This shows all 226 secrets grouped by tier with their rotation method.
Secrets marked `[SKIP]` require manual rotation or are regenerated by
their convergence role.

### Step 1: Rotate Tier 1 (Critical — Directly Exposed)

```bash
# Plan
python scripts/emergency_credential_rotation.py --tier t1 --plan

# Apply
python scripts/emergency_credential_rotation.py --tier t1 --apply
```

Immediately reconverge Semaphore:
```bash
make converge-semaphore env=production
```

Separately revoke the Gitea API token:
1. Log in to Gitea at `https://git.lv3.org` as admin
2. Go to `ops-gitea` user settings > Applications
3. Delete the old/compromised token
4. Generate a new token
5. Update the git remote: `git remote set-url gitea https://ops-gitea:<NEW_TOKEN>@git.lv3.org/ops/proxmox_florin_server.git`

## Receipts

Every `--apply` run writes a timestamped receipt to:
```
receipts/credential-rotations/<timestamp>-emergency-rotation-<tier>.json
```

These receipts contain:
- Which secrets were rotated, skipped, or errored
- The generation method used (not the value itself)
- Timestamp for audit trail

## Troubleshooting

**Service fails to start after rotation:**
The most common cause is convergence order — the service tried to connect to
PostgreSQL with the new password before the Postgres role was reconverged.
Fix: reconverge `postgres-vm` first, then the failing service.

**Keycloak OIDC login broken:**
The client secret was rotated but the service hasn't been reconverged yet.
Fix: reconverge the specific service that's failing OIDC.

**OpenBao sealed after rotation:**
The `openbao_init_payload` was intentionally skipped. If OpenBao is sealed,
use the existing unseal keys from `.local/openbao/init.json`.

**SSH access lost after key rotation:**
If the bootstrap key was rotated but not yet deployed, use the Proxmox console
(web UI at `https://proxmox.lv3.org:8006`) to access VMs directly and deploy
the new public key.
