# ADR 0138: Published Artifact Secret Scanning

- Status: Proposed
- Implementation Status: Not Implemented
- Implemented In Repo Version: not yet
- Implemented In Platform Version: not yet
- Implemented On: not yet
- Date: 2026-03-24

## Context

The platform already runs `gitleaks` as a pre-commit hook to detect secrets in committed source files. This is the right control for source code and configuration files. However, there are several artifact types that are generated at runtime and published to portals or committed to the repository that are not covered by the existing git pre-commit scan:

1. **Live-apply receipts** (`receipts/*.receipt.json`): Generated by Windmill after each workflow run. May contain workflow output that includes command output, ansible task results, or API responses that inadvertently echo secrets.

2. **Triage reports** (`.local/triage/reports/*.json`): Generated by the triage engine (ADR 0114). May include log snippets or signal values that contain credential strings from error messages.

3. **Changelog portal renders**: The changelog portal (ADR 0081) renders ledger event summaries. A ledger entry's `payload` field may contain a secret if a workflow incorrectly logs credential material.

4. **ADR and runbook documents**: While ADRs should never contain secrets, they sometimes include example commands with placeholder values that look like real secrets to pattern matchers (e.g., `export API_TOKEN=example-token-here`).

5. **Search index documents**: The search fabric (ADR 0121) indexes receipts, cases, and log summaries. If a secret reaches the index, it becomes searchable by any authenticated operator.

6. **Agent state store entries** (ADR 0130): Agents might checkpoint sensitive API responses into the state store. The state store itself is not indexed, but its content could be exported by a workflow.

The existing `gitleaks` pre-commit hook runs on staged files at commit time. It does not scan:
- Runtime-generated files that are committed as part of automated workflows.
- Postgres-stored data (ledger, state store, search index).
- The rendered output of the changelog and docs portals.

A secret that enters the system through a workflow output or error message can persist in the ledger, be indexed by search, and be rendered in the changelog portal — all without ever being detected by `gitleaks`.

## Decision

We will implement a **multi-stage secret scanning pipeline** that covers runtime-generated artifacts in addition to the existing pre-commit source scan.

### Stage 1: Pre-publish receipt scan (pre-commit hook extension)

The existing `.pre-commit-config.yaml` will be extended to scan receipt files with `gitleaks` in addition to source files:

```yaml
- repo: local
  hooks:
    - id: scan-receipts
      name: Secret scan live-apply receipts
      entry: gitleaks detect --source receipts/ --config .gitleaks.toml --no-git
      language: system
      files: ^receipts/
      pass_filenames: false
```

Any receipt file that triggers a secret match fails the pre-commit hook. The receipt must be redacted (ADR 0134) before being committed.

### Stage 2: Post-workflow ledger entry scan

After every Windmill workflow completes, a Windmill sidecar workflow `scan-ledger-entry` is triggered by a NATS subscription on `platform.execution.completed` (ADR 0124). It reads the most recent ledger entry for the completed job and runs `gitleaks` against the `payload` JSON field:

```python
# config/windmill/scripts/scan-ledger-entry.py

def scan_ledger_entry(event: dict):
    entry = ledger.read(event["intent_id"])
    result = gitleaks_scan_string(json.dumps(entry.payload))
    if result.findings:
        ledger.write(
            event_type="security.secret_detected_in_ledger",
            target_id=event["intent_id"],
            metadata={
                "findings": result.findings,
                "action": "payload_redacted",
            }
        )
        # Redact the payload in-place (only allowed mutation to a ledger entry)
        ledger.redact_payload(event["intent_id"], redaction_token="[REDACTED BY SECRET SCANNER]")
        nats.publish("platform.security.secret_detected", {...})
        mattermost.post("#platform-security", f"⚠️ Secret detected in ledger entry {event['intent_id']}")
```

Ledger entries are normally append-only, but secret detection is the one permitted mutation: the payload field is replaced with `[REDACTED BY SECRET SCANNER]` and the original entry is preserved in a separate `ledger.redacted_events` table accessible only to `platform-admin` role (ADR 0056).

### Stage 3: Search index scan (nightly)

A nightly Windmill workflow `scan-search-index` runs `gitleaks` against the full text of all `search.documents` entries written in the last 24 hours. Any match triggers redaction of the `body` field in the search index and a security ledger event.

### Stage 4: Triage report scan

The triage engine (ADR 0114) runs `gitleaks` against each generated triage report before writing it to disk or the ledger. Reports with matches are written with the offending section replaced before any downstream processing.

### `.gitleaks.toml` configuration

The existing `.gitleaks.toml` is extended with rules for infrastructure-specific patterns:

```toml
# .gitleaks.toml additions

[[rules]]
id = "openbao-token"
description = "OpenBao / Vault API token"
regex = '''hvs\.[A-Za-z0-9_-]{20,}'''
tags = ["openbao", "vault"]

[[rules]]
id = "step-ca-provisioner-password"
description = "Step-CA provisioner password (typical pattern in configs)"
regex = '''provisioner[_\-]?password["\s:=]+[A-Za-z0-9+/]{20,}'''
tags = ["step-ca", "pki"]

[[rules]]
id = "keycloak-client-secret"
description = "Keycloak client secret (UUID format in config)"
regex = '''client.secret["\s:=]+[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}'''
tags = ["keycloak"]

[allowlist]
paths = [
  ".gitleaks.toml",           # Rule definitions contain patterns
  "docs/adr/",                # ADRs contain example tokens that are not real
  "tests/fixtures/",          # Test fixtures may contain fake token patterns
]
regexes = [
  "example-token-here",       # Explicit placeholder strings
  "\\[REDACTED",              # Already-redacted placeholders
  "YOUR_.*_HERE",             # Template placeholder pattern
]
```

### Alerting on detection

Any detection triggers:
1. Immediate Mattermost post to `#platform-security`.
2. A `security.secret_detected` ledger event (high-severity).
3. A GlitchTip issue opened with the detection details.
4. The triage engine is invoked with a pre-filled `secret_detected` alert payload.

## Consequences

**Positive**

- Secrets that enter via workflow output, error messages, or API responses are caught at the ledger layer before they can propagate to the search index or portal renderer.
- The nightly index scan provides a backstop for any secrets that entered the index before the Stage 2 scan was deployed.
- Platform-specific token patterns (OpenBao, step-ca, Keycloak) are explicitly detected rather than relying solely on generic entropy-based detection.

**Negative / Trade-offs**

- The post-workflow ledger scan adds latency to every workflow completion. If `gitleaks` takes 200ms per ledger entry, a platform that runs 100 workflows per day adds 20 seconds of scanning overhead. This is acceptable but should be measured.
- Ledger payload redaction (the one permitted mutation to ledger entries) creates a distinction between the audit record and the displayed record. Post-incident analysis must account for redacted entries and retrieve the original from `ledger.redacted_events` with `platform-admin` access.
- False positives in ADRs and runbooks (example tokens, placeholder values) must be maintained in the `allowlist` section. An over-aggressive allowlist will miss real secrets.

## Boundaries

- This ADR covers runtime-generated text artifacts. It does not cover binary artifacts (container images, Packer build artifacts, backup archives) — those have separate supply chain controls (ADR 0068).
- Scanning is advisory for ADR/runbook documents (they are in the `allowlist`). The author is responsible for not including real credentials in documentation.
- Secret revocation (rotating a leaked secret) is the operator's responsibility once detected; this ADR only handles detection and redaction of the stored artifact.
