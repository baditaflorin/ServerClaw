# ADR 0093: Interactive Ops Portal with Live Actions

- Status: Accepted
- Implementation Status: Implemented
- Implemented In Repo Version: 0.97.0
- Implemented In Platform Version: 0.105.0
- Implemented On: 2026-03-24
- Date: 2026-03-23

## Context

The current ops portal (ADR 0074) is a static HTML page generated by `scripts/generate_ops_portal.py` and published at `ops.example.com`. It is read-only: it shows service status, recent deployments, and platform topology, but an operator who spots a problem must leave the browser, open a terminal, locate the right `make` target or `lv3` CLI command, and run it. The portal is a dashboard, not a control plane.

This creates friction at exactly the wrong moment — during an incident or a deployment. Operators context-switch between the browser (visibility) and the terminal (action) repeatedly. Ad-hoc terminal sessions also bypass the mutation audit log (ADR 0066) and the maintenance window protocol (ADR 0080) unless the operator consciously uses the `lv3` CLI.

The platform already has Windmill (ADR 0044) as an orchestration engine with a workflow API, the platform API gateway (ADR 0092) as an authenticated action surface, NATS (ADR 0058) for real-time event streaming, and Keycloak (ADR 0056) for SSO. The static portal could be upgraded to use all of these.

The non-goal is building a general-purpose infrastructure UI from scratch. Portainer (ADR 0055) handles Docker operations; the Proxmox UI handles VM operations. The ops portal's scope is **platform-level operations**: deploy a service, rotate a secret, acknowledge an alert, trigger a runbook, view drift — actions that span multiple underlying systems.

## Decision

We will replace the static generated ops portal with a **server-side-rendered interactive portal** built with FastHTML (Python) or HTMX-augmented Jinja2 templates, backed by the platform API gateway (ADR 0092), and deployed as a Compose service on `docker-runtime`.

### Technology choice

**FastHTML** (Python HTMX framework) is chosen over React/Vue because:
- The platform is Python-native; no JavaScript build toolchain is introduced
- FastHTML renders HTML server-side and uses HTMX for interactivity; no SPA complexity
- It integrates naturally with the existing FastAPI gateway via `httpx` calls
- The portal never stores state; it is a thin rendering layer over the gateway API

### Portal sections

#### 1. Platform Overview

Real-time status panel populated from `/v1/platform/health`:
- Service health traffic lights (green/yellow/red) for every registered service
- Last deployment timestamp per service
- Drift status indicator (from ADR 0091): green = clean, yellow = warn drift, red = critical drift
- Active maintenance windows (from ADR 0080)

The panel auto-refreshes every 30 seconds via HTMX `hx-trigger="every 30s"`.

#### 2. Service Actions

Each service card in the portal includes action buttons:

| Button | Action | Confirmation required |
|---|---|---|
| Deploy | POST `/v1/platform/deploy` → triggers Windmill workflow | Yes — shows diff preview |
| Health check | GET `/v1/health/<service>` | No |
| Rotate secret | POST `/v1/platform/secrets/rotate` | Yes — shows affected services |
| View logs | Opens Grafana/Loki deep-link for service | No |
| Restart | POST `/v1/platform/deploy` with `restart_only: true` | Yes |

All actions emit a NATS event and are recorded in the mutation audit log (ADR 0066) with the Keycloak identity of the operator who clicked the button.

#### 3. Deployment Console

A live log stream panel that subscribes to NATS subject `platform.deploy.*` and renders lines as they arrive via server-sent events (SSE). When an operator triggers a deployment, they can watch the Ansible/Windmill output in the browser without opening a terminal.

#### 4. Drift Panel

Renders the latest drift report from `/v1/platform/drift`. Each drift item has:
- Service name, drift type, severity
- "Reconcile" button → POST `/v1/platform/deploy` for that service
- Link to the relevant ADR

#### 5. Runbook Launcher

A catalogue of operator runbooks (Windmill workflows) with a one-click launch button and a parameter form rendered from the Windmill workflow schema. Runbooks include: rotate all certificates, re-apply a specific service, run a health probe sweep, trigger backup restore verification (ADR 0099).

#### 6. Changelog

The deployment history from `scripts/deployment_history.py`, rendered as a reverse-chronological timeline with links to receipts.

### Authentication

The portal is protected by Keycloak OIDC via the nginx edge (ADR 0056 + ADR 0021). The portal itself exchanges the Keycloak session token for a short-lived API token from the gateway. All downstream API calls carry this token, so the audit log captures the operator's Keycloak identity — not a service account.

### Compose service

```yaml
services:
  ops-portal:
    image: registry.example.com/platform/ops-portal:latest
    ports:
      - "8092:8090"
    environment:
      GATEWAY_URL: "http://api-gateway:8080"
      KEYCLOAK_URL: "https://sso.example.com"
      KEYCLOAK_REALM: "lv3"
      KEYCLOAK_CLIENT_ID: "ops-portal"
      NATS_URL: "nats://nats:4222"
    healthcheck:
      test: ["CMD", "curl", "-sf", "http://localhost:8092/health"]
      interval: 30s
      timeout: 5s
      retries: 3
```

Published at `ops.example.com` via nginx edge with OIDC enforcement (replacing the current static file serving).

### Transition from static portal

The existing `scripts/generate_ops_portal.py` continues to run and writes its output to `receipts/ops-portal-snapshot.html` as a point-in-time read-only archive. The live portal at `ops.example.com` switches to the dynamic version. This is a non-breaking change: the DNS entry and OIDC client already exist.

### Repository implementation

The repository implementation delivered in `0.97.0` includes:

- `scripts/ops_portal/` — FastAPI + Jinja templates for the interactive portal shell, live action surface, drift panel, runbook launcher, changelog panel, and SSE deployment console
- `roles/ops_portal_runtime/` plus `playbooks/ops-portal.yml` — container build and runtime deployment on `docker-runtime`
- public-edge OIDC propagation updates so the protected proxy forwards the authenticated operator identity and access token to the portal, while `GET /health` remains unauthenticated
- catalog updates for the interactive runtime contract in `config/service-capability-catalog.json`, `config/health-probe-catalog.json`, `config/subdomain-catalog.json`, and `config/api-gateway-catalog.json`
- legacy static snapshot archiving through `receipts/ops-portal-snapshot.html`

The live cutover from the existing static portal was completed on 2026-03-24, including the `8092` runtime publish correction, guest firewall synchronization, and public edge proxy update on the current `0.105.0` platform release.

## Consequences

**Positive**
- Platform operations become a browser-native experience; no terminal required for routine operations
- All actions flow through the API gateway and are captured in the mutation audit log — improved operational accountability
- Deployment log streaming eliminates the "is this running?" uncertainty during deployments
- The portal is the single operator entry point: from here, all platform services, actions, and runbooks are reachable
- FastHTML keeps the codebase Python-only with no frontend build pipeline

**Negative / Trade-offs**
- The portal is now a service with a runtime dependency on the API gateway and NATS; it must be monitored and has a startup order dependency
- FastHTML is a young framework (2024-era); it has fewer resources and community examples than React; any non-trivial UI pattern requires reading its source
- SSE-based log streaming requires persistent HTTP connections; Caddy/nginx must be configured with appropriate proxy timeouts (`proxy_read_timeout 300s`)

## Alternatives Considered

- **Enhance the static portal with JavaScript fetch calls**: avoids a new service but introduces unstructured JavaScript that is not type-checked or testable; grows into an unmaintainable tangle quickly
- **Use Grafana as the ops portal**: Grafana is already in the stack (ADR 0011); it handles metrics and logs well but is not suited for triggering Ansible playbooks, rendering dynamic forms, or audit-logging operator clicks
- **Windmill as the ops portal**: Windmill's UI can trigger workflows; but it exposes all workflows equally, has no platform-specific layout, and does not render the topology/drift/changelog context that makes the ops portal useful
