# ADR 0143: Gitea for Self-Hosted Git and Webhook-Driven Automation

- Status: Proposed
- Implementation Status: Not Implemented
- Implemented In Repo Version: not yet
- Implemented In Platform Version: not yet
- Implemented On: not yet
- Date: 2026-03-24

## Context

The platform repository (`proxmox_florin_server`) is currently hosted on an external git provider. This creates several dependencies:

- The ADR indexer (ADR 0121) uses a webhook from the external provider. The webhook secret is a long-lived credential stored in the platform and in the provider, creating a shared secret that is difficult to rotate.
- The pre-push validation gate (ADR 0031) runs locally on the operator's machine. There is no server-side enforcement of the validation pipeline; a force-push bypasses it.
- The search fabric indexer (ADR 0121), changelog generator (ADR 0081), and developer portal builder (ADR 0094) all trigger from git push events. These triggers depend on the external provider's webhook delivery, which introduces latency and availability coupling.
- There is no local CI for integration tests (ADR 0111). Tests that require platform API calls cannot run in an external CI context without exposing the platform API externally.
- The external provider logs every git operation, creating a record of deployment timing and commit metadata outside the operator's control.

A self-hosted Gitea instance resolves all of these: local webhook delivery with no external dependencies, server-side pre-receive hooks for validation gate enforcement, local Gitea Actions CI with direct access to platform APIs, and no external metadata exposure.

## Decision

We will deploy **Gitea** as a self-hosted git service on the `docker-runtime-lv3` VM, replacing the external git provider as the primary remote.

### Deployment

```yaml
# In versions/stack.yaml — new service entry
- service: gitea
  vm: docker-runtime-lv3
  image: gitea/gitea:latest
  port: 3030
  data_volume: /data/gitea
  keycloak_oidc: true
  access: tailscale_only
  subdomain: git.lv3.org   # Private, Tailscale-only; not published to public internet
```

Gitea is deployed as a Docker Compose service with:
- Postgres `gitea` database (on `postgres-lv3`).
- Keycloak OIDC integration via the platform SSO (ADR 0056): operator login via `sso.lv3.org`.
- Storage on a named Docker volume backed by the PBS backup policy (ADR 0029).
- `access: tailscale_only` — Gitea is never exposed publicly. The operator's machine accesses it via Tailscale.

### Gitea Actions CI

Gitea's built-in Actions system (compatible with GitHub Actions syntax) provides local CI without external dependencies. A self-hosted runner is deployed on `docker-build-lv3`:

```yaml
# .gitea/workflows/validate.yml
on: [push]
jobs:
  validate:
    runs-on: self-hosted
    steps:
      - uses: actions/checkout@v4
      - name: Validate data models
        run: make validate-data-models
      - name: Validate YAML/JSON
        run: pre-commit run --all-files
      - name: Run integration tests
        run: pytest tests/ -m "not live"  # Exclude tests that require live platform access
```

### Pre-receive hook for server-side validation

Gitea supports pre-receive hooks. The repository validation pipeline (ADR 0031) is registered as a pre-receive hook that runs `make validate-data-models` on every push:

```bash
#!/bin/bash
# /data/gitea/gitea/repositories/ops/proxmox_florin_server.git/hooks/pre-receive
while read oldrev newrev refname; do
    git checkout $newrev -- .
    make validate-data-models 2>&1
    if [ $? -ne 0 ]; then
        echo "Repository validation failed. Push rejected."
        exit 1
    fi
done
```

This enforces the validation pipeline even for pushes from the CLI, preventing a `--no-verify` bypass.

### Webhook-driven platform automation

Gitea webhooks replace the external provider webhooks:

| Event | Webhook target | Effect |
|---|---|---|
| `push` to `main` | `windmill/search-index-rebuild` | Re-indexes ADRs, runbooks, configs (ADR 0121) |
| `push` to `main` | `windmill/generate-changelog-portal` | Rebuilds changelog site (ADR 0081) |
| `push` to `main` | `windmill/generate-docs-portal` | Rebuilds developer portal (ADR 0094) |
| `push` to `main` | `windmill/generate-platform-manifest` | Refreshes platform manifest (ADR 0132) |

Webhook delivery is local (Gitea → Windmill on the same Docker network), eliminating external dependency and latency.

### API integration for agents

Gitea's API is used by agents for:
- Checking branch status before submitting an intent that modifies the repository.
- Creating pull requests for ADR drafts generated by Claude Code sessions.
- Querying commit history as a signal in the triage engine (ADR 0114): "was there a push to main in the last 2 hours?" without depending on the external provider.

```python
# platform/git/client.py
gitea = GiteaClient(base_url="http://gitea:3030", token=openbao.get("gitea/platform-agent-token"))
recent_commits = gitea.get_commits(repo="ops/proxmox_florin_server", since=datetime.now() - timedelta(hours=2))
```

### Mirror to external provider (optional)

For disaster recovery and external collaboration, a Gitea mirror job pushes to the external provider after each successful push to local Gitea. This is push-only; the external provider is no longer the source of truth.

## Consequences

**Positive**

- Local webhook delivery eliminates external availability coupling. Platform automation (search indexing, changelog rebuild) is not dependent on the external provider's webhook delivery infrastructure.
- Server-side pre-receive validation gates are now enforceable. A push that would fail the validation pipeline is rejected at the server, regardless of whether the operator ran pre-commit locally.
- Local CI (Gitea Actions) can run tests that call the platform API directly without exposing the API publicly.
- All git operations are private: the external provider no longer sees commit metadata, push timing, or branch history.

**Negative / Trade-offs**

- Gitea is an additional service to operate and maintain. It requires backups, updates, and monitoring like any other platform service.
- Migrating the existing repository from the external provider requires a one-time migration run and coordination to update all developer clones to use the new remote.
- Gitea Actions, while GitHub Actions compatible, has gaps in the ecosystem (fewer third-party actions, some actions require the full GitHub Actions runner which is different from the Gitea runner).

## Boundaries

- Gitea is Tailscale-only. It is never published to the public internet. External contributors (if any) must be onboarded via Tailscale access.
- This ADR does not govern how ADRs are written or merged; it only provides the self-hosted infrastructure for the existing workflow.
