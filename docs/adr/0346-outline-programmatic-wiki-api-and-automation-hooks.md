# ADR 0346: Outline Programmatic Wiki API and Automation Hooks

- Status: Accepted
- Implementation Status: Implemented
- Implemented In Repo Version: 0.178.4
- Implemented In Platform Version: 0.178.4
- Implemented On: 2026-04-05
- Date: 2026-04-05

## Context

ADR 0199 established Outline as the living knowledge wiki and wired it to the repo via `sync_docs_to_outline.py`. That script is designed for operator-initiated, batch sync of collection landing pages from repo markdown.

Two capabilities are now missing:

1. **Agent-facing programmatic API** — other LLM agents, CI pipelines, and automation scripts cannot publish or manage wiki content without understanding the Outline HTTP API internals. There is no governed, repo-managed entry point for one-off document writes, collection management, or user access control.

2. **Automation hooks** — high-value platform events (releases, validation gate runs, nightly integration tests, mutation audit summaries, DR status) are generated by existing automation but never land in the wiki. Operators and agents must track findings across multiple disparate sinks (ntfy, NATS, JSONL files, GitHub Actions artifacts) rather than having them searchable in one place.

## Decision

### 1. Shared API client module — `scripts/outline_client.py`

Extract the `OutlineClient`, `OutlineError`, and API helper functions (`collections_by_name`, `documents_in_collection`, `ensure_document`, `load_api_token`) into a shared module imported by both `sync_docs_to_outline.py` and the new agent tool. This prevents code duplication across the two entry points.

### 2. Agent-facing CLI tool — `scripts/outline_tool.py`

A new stdlib-only Python script that exposes every routine wiki operation as a subcommand with JSON stdout output. All agents and automation scripts should use this tool exclusively rather than calling the Outline HTTP API directly.

**Document commands:**
```
document.publish  --collection NAME --title TITLE (--file PATH | --stdin) [--parent TITLE] [--rewrite-links]
document.get      --collection NAME --title TITLE
document.list     --collection NAME
document.delete   --collection NAME --title TITLE
document.search   --query TEXT [--collection NAME]
```

**Collection commands:**
```
collection.list
collection.create   --name NAME [--description TEXT]
collection.delete   --name NAME
collection.members  --collection NAME
collection.grant    --collection NAME --user EMAIL_OR_NAME --permission read|read_write
collection.revoke   --collection NAME --user EMAIL_OR_NAME
collection.grant-group  --collection NAME --group NAME --permission read|read_write
collection.revoke-group --collection NAME --group NAME
```

**User and group commands:**
```
user.list
group.list
group.create  --name NAME
group.delete  --name NAME
group.members --group NAME
group.add-user    --group NAME --user EMAIL_OR_NAME
group.remove-user --group NAME --user EMAIL_OR_NAME
```

**Convenience commands:**
```
changelog.push  [--file PATH] [--title TITLE]
```

All commands accept `--token`, `--token-file`, and `OUTLINE_API_TOKEN` env var for auth. All output is newline-terminated JSON with an `"ok"` boolean so callers can detect failures without exit-code parsing:
```json
{"ok": true, "outcome": "created", "title": "...", "url": "/doc/..."}
{"ok": false, "error": "collection not found: X"}
```

### 3. New managed collections

Two new collections are bootstrapped and maintained by `sync_docs_to_outline.py sync`:

- **Changelogs** — release notes, version history, and the full `changelog.md`. Agents and the release manager push here after each release via `changelog.push`.
- **Automation Runs** — per-run summaries from live playbook runs, CI/CD pipelines, nightly integration tests, and fault injection jobs.

All collections are created with `permission: "read"` and `sharing: True` so every authenticated workspace member can browse and search without explicit grants.

### 4. Link rewriting

Repository markdown contains relative file links (`docs/release-notes/0.178.3.md`) that break in the wiki context. The tool rewrites relative links to absolute GitHub URLs (`https://github.com/baditaflorin/proxmox_florin_server/blob/main/...`) before publishing. Links to `build/` artifacts and `.local/` paths are stripped to plain text. The `--rewrite-links` flag on `document.publish` enables this for ad-hoc publishes; `changelog.push` always rewrites.

### 5. API token scope expansion

The `outline.automation` API token must be bootstrapped with expanded scopes to unlock user and group management:

```
collections.memberships  collections.addUser    collections.removeUser
collections.addGroup     collections.removeGroup
groups.create  groups.delete  groups.info  groups.list  groups.update
groups.memberships  groups.addUser  groups.removeUser
users.list  users.read
```

The canonical scope list lives in `API_TOKEN_SCOPES` in `sync_docs_to_outline.py` and is mirrored in `outline_api_token_scopes` in the Ansible role defaults. Re-bootstrap by deleting `.local/outline/api-token.txt` and running the `bootstrap-token` subcommand.

### 6. Automation hooks

The following platform events are wired to publish into the wiki automatically:

| Trigger | Collection | Document title pattern | How |
|---|---|---|---|
| Release merge | Changelogs | `v{version}` | `release_manager.py` calls `changelog.push --title v{version}` |
| Release notes file | Changelogs | `Release Notes {version}` | `release_manager.py` publishes individual release note files |
| Gitea CI `validate` job | Automation Runs | `ci-validate-{branch}-{sha7}` | Post-step in `.gitea/workflows/validate.yml` |
| Post-merge gate (Windmill) | Automation Runs | `post-merge-gate-{date}` | `config/windmill/scripts/post-merge-gate.py` |
| Nightly integration tests | Automation Runs | `nightly-tests-{date}` | `config/windmill/scripts/nightly-integration-tests.py` |
| Ansible live apply | Automation Runs | `live-apply-{playbook}-{date}` | `outline_runtime` publish task |
| `make gate-status` | Automation Runs | `gate-status-{date}` | Makefile target post-step |

All hooks use `outline_tool.py document.publish --stdin` so they can be triggered with a simple pipe from any script that produces markdown or plain text.

### 7. Default permissions model

Collections and documents are **workspace-readable by default**. This is appropriate because the wiki mirrors content that is already public in the repository. Restricted access is opt-in via `collection.grant` or by omitting `permission: "read"` when creating a collection.

## Consequences

- Other LLM agents working in this repository can call `scripts/outline_tool.py` to publish findings, run summaries, and investigation notes without needing Outline API knowledge.
- Platform events that previously existed only as ephemeral ntfy notifications or local JSONL files now accumulate as searchable wiki documents.
- The release workflow gains a one-command changelog publication step.
- `sync_docs_to_outline.py` remains the authoritative sync path for repo-managed collection landing pages; `outline_tool.py` handles event-driven and agent-initiated writes.
- User and group management is available once the token is re-bootstrapped with the expanded scope list.
- ADR 0199 remains in force; this ADR extends it with the programmatic publication layer.
