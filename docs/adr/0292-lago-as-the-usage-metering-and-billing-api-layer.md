# ADR 0292: Lago As The Usage Metering And Billing API Layer

- Status: Accepted
- Implementation Status: Implemented
- Implemented In Repo Version: 0.177.118
- Implemented In Platform Version: 0.130.77
- Date: 2026-03-29
- Implemented On: 2026-03-31

## Context

Platform services that consume resources on behalf of tenants—LLM inference
calls, document processing jobs, storage consumed, API calls billed to
external partners—currently have no unified mechanism to:

- ingest usage events from multiple sources and aggregate them into
  billable quantities
- define billing plans (per-seat, per-event, per-GB, graduated pricing)
  in a single place and apply them to subscriptions
- generate invoices or usage summaries that can be exported to an
  accounting system or sent to a tenant via API

Usage tracking is currently done by counting rows in application-specific
PostgreSQL tables and summing them with ad-hoc SQL. There is no API that
accepts a usage event from a service and returns the current period's
aggregated consumption for a given metric. Generating a cost breakdown
requires a manual SQL query session.

Lago is a CPU-only, open-source usage-based billing and metering platform.
It exposes a complete REST API—event ingestion, subscription management, plan
definition, invoice generation, and credit note creation—all documented with
a published OpenAPI specification. The service accepts events from any
producer via HTTP POST and aggregates them according to the billable metric
definition without the producer needing to know the billing logic.

## Decision

We will deploy **Lago** as the usage metering and billing API layer for
platform services that track resource consumption or bill external tenants.

### Deployment rules

- Lago runs as a Docker Compose stack (API, worker, Redis, and ClickHouse)
  on the docker-runtime VM using the official `getlago/lago` images
- Authentication to the Lago UI and management API is delegated to Keycloak
  via OIDC (ADR 0063)
- The service is published under the platform subdomain model (ADR 0021) at
  `billing.<domain>`; the REST API is at `billing.<domain>/api/v1/`
- Lago uses the shared PostgreSQL cluster (ADR 0042) with a dedicated `lago`
  database; ClickHouse is deployed as a Lago-internal sidecar for event
  aggregation queries
- All persistent state is stored on named Docker volumes included in the
  backup scope (ADR 0086)
- Secrets (database credentials, OIDC client credentials, Lago secret key)
  are injected from OpenBao following ADR 0077

### API-first operation rules

- All billable metric definitions, plans, coupons, and tax rates are
  created via the Lago REST API; the Ansible role declares the initial
  catalogue in `defaults/main.yml` and applies it idempotently via the
  API seed task on every converge
- Usage events are submitted exclusively via the Lago event ingest API
  (`POST /api/v1/events`); services do not write usage records directly
  to PostgreSQL or ClickHouse
- Event submission is asynchronous and non-blocking from the producer's
  perspective; the Lago worker processes ingested events and applies
  aggregation rules independently of the producer's request lifecycle
- Invoice generation, subscription upgrades, and credit note creation are
  performed via the Lago REST API from Windmill billing automation flows;
  no GUI steps are part of the standard billing cycle
- The Lago API key for each producing service is stored in OpenBao and
  retrieved at service startup; it is scoped to event ingest only and
  cannot read subscription or invoice data

### Event schema rules

- all events sent to the Lago ingest API include:
  - `transaction_id` — a UUID generated by the producer, used for
    idempotent deduplication
  - `external_subscription_id` — the identifier of the subscribing tenant
    or service
  - `code` — the billable metric code (e.g. `llm_tokens`, `storage_gb`,
    `api_calls`)
  - `timestamp` — ISO 8601 UTC timestamp of when the resource was consumed
  - `properties` — a JSON object with metric-specific fields (e.g.
    `{"input_tokens": 512, "output_tokens": 128}`)
- events that fail the Lago schema validation are published to NATS
  JetStream (ADR 0276) on the `billing.events.rejected` topic for
  operator review

## Consequences

**Positive**

- Any platform service becomes a metered producer with a single HTTP POST;
  it does not need to know billing plan logic, aggregation windows, or
  invoice generation rules.
- The Lago API provides a queryable current-period usage summary for any
  subscription; Windmill flows can surface this to operators or tenants
  without SQL queries.
- Billable metric definitions and plans are version-controlled in the
  Ansible role; the billing configuration is reproducible after a fresh
  deploy.
- Lago's idempotent event ingest (via `transaction_id`) means network
  retries from producers do not double-count usage.

**Negative / Trade-offs**

- Lago's ClickHouse sidecar adds approximately 400 MB of baseline memory;
  the docker-runtime VM must have headroom alongside Plausible's ClickHouse
  instance. Running two ClickHouse instances is wasteful; a future
  consolidation to a shared ClickHouse service should be evaluated.
- Lago's billing cycle and invoicing concepts assume a SaaS tenancy model;
  adapting them to internal cost allocation (chargeback between platform
  teams) requires mapping internal consumers to Lago's "customer" concept.

## Boundaries

- Lago meters usage and generates billing artefacts; it does not process
  payments. Payment collection is out of scope for the self-hosted platform.
- Lago does not replace Grafana for infrastructure cost dashboards; Lago
  handles billable-metric aggregation at the tenant level, while Grafana
  handles infrastructure resource utilisation at the host level.
- Lago events represent resource consumption events, not application logs;
  application logging remains in Loki.
- The Lago UI is available for plan and subscription management; routine
  operations performed through the UI are treated as drift if they modify
  the metric and plan catalogue declared in the Ansible role.
