# ADR 0386: Agent Cross-Host Execution and MCP Tool Surface

- Status: Accepted
- Implementation Status: Partial
- Implemented In Repo Version: 0.178.70
- Implemented In Platform Version: 0.178.70
- Implemented On: 2026-04-09
- Date: 2026-04-09

## Context

LLM agents (ServerClaw via chat.lv3.org, Dify workflows, Claude Code sessions,
MCP-connected tools) need to inspect and operate the platform's physical hosts
— not just read committed files or call HTTP APIs.

### What exists today

ADR 0069 established the agent tool registry with 37 governed tools.  ADR 0125
defines capability bounds per identity.  ADR 0227 provides bounded command
execution via systemd-run wrappers.  ADR 0369 standardises the HTTP gateway
pattern for service-level API tools (Plane, Outline, Nomad).

These work well for:
- **Observe**: reading platform state, receipts, health probes
- **Execute (single host)**: governed commands on the API gateway host

They do **not** cover:
1. **True host shell access** — agents need `systemctl`, `journalctl`, `docker`,
   `free`, `df`, `ip addr` — the host's own binaries, not a sandboxed container
2. **Multi-host execution** — the platform spans 7+ VMs; agents need to run
   commands on any of them, not just runtime-control-lv3
3. **MCP-native tool discovery** — chat surfaces (LibreChat, Claude Code) use
   MCP; tools should be discoverable and callable without HTTP proxy wiring
4. **Audit and traceability** — every agent command must be logged with identity,
   timestamp, command, host, exit code, and output hash — regardless of which
   surface initiated it (chat, API, MCP, Dify workflow)
5. **SSH key lifecycle** — the API gateway host currently has no SSH keys to
   reach other VMs; a managed key pair with automatic rotation is needed

### The problem with the previous approach

The initial `execute-host-command` implementation (v0.178.69) created a temporary
Docker container with the host root mounted read-only at `/host`.  This was
**sandboxed** — commands ran inside a minimal `debian:bookworm-slim` image, not
the host's own rootfs.  `free`, `systemctl`, `docker ps` all failed because the
container image lacked those binaries.

## Decision

### 1. nsenter for true host shell access

Local commands use `nsenter --target 1 --mount --uts --ipc --net --pid` inside
a privileged temporary container to enter the host's PID 1 namespaces.  This
gives commands access to the host's own binaries, filesystems, systemd, Docker
daemon, and network stack — identical to running as root on the host.

The temporary container is:
- Created via Docker Engine API (Unix socket)
- Uses `alpine:3.20` (tiny base; nsenter is a busybox builtin)
- Runs with `Privileged: true` and `PidMode: host`
- Automatically removed after execution

### 2. Multi-host execution via SSH

For remote hosts, the nsenter container runs:
```
nsenter --target 1 ... -- ssh -o StrictHostKeyChecking=no root@<target_ip> <command>
```

This uses the host's own SSH client and keys (not the container's).

Supported targets with friendly names:

| Name | IP | Purpose |
|------|----|---------|
| `runtime-control-lv3` | local | API gateway, agent tools, control plane |
| `proxmox` | 10.10.10.1 | Hypervisor |
| `postgres-lv3` | 10.10.10.60 | Shared PostgreSQL |
| `docker-runtime-lv3` | 10.10.10.20 | Docker workloads, Dify, LibreChat |
| `build-server` | 10.10.10.30 | CI/CD, gate runner |
| `coolify-lv3` | 10.10.10.70 | Coolify PaaS |
| `runtime-comms-lv3` | 10.10.10.21 | Neko, LiveKit |

### 3. SSH key provisioning (Phase 2 — not yet implemented)

The API gateway host needs a dedicated SSH key pair for cross-host execution:

```yaml
# roles/api_gateway_runtime/tasks/ssh-agent-key.yml
- name: Generate agent SSH key pair
  community.crypto.openssh_keypair:
    path: /etc/lv3/api-gateway/agent-ssh-key
    type: ed25519
    comment: "agent-exec@runtime-control-lv3"
    owner: root
    mode: "0600"

- name: Distribute public key to all platform hosts
  ansible.posix.authorized_key:
    user: ops
    key: "{{ agent_ssh_public_key }}"
    comment: "agent-exec@runtime-control-lv3 (ADR 0386)"
  delegate_to: "{{ item }}"
  loop: "{{ groups['all'] }}"
```

Key lifecycle:
- Generated by Ansible on first convergence
- Public key distributed to all managed hosts via `authorized_key`
- Stored at `/etc/lv3/api-gateway/agent-ssh-key` (not in `.local/`)
- Rotation: regenerate and redistribute via `make converge-api-gateway`
- Audit: key fingerprint logged in live-apply receipt

### 4. Safety boundaries

**Blocked patterns** — commands containing these substrings are rejected before
execution, regardless of host:

```python
_HOST_COMMAND_BLOCKED_PATTERNS = [
    "rm -rf /", "mkfs", "dd if=/dev/zero", "dd if=/dev/urandom",
    ":(){ :", "> /dev/sd", "shutdown", "reboot", "init 0", "init 6",
    "passwd", "userdel",
]
```

**Capability bounds** — the `execute-host-command` tool respects ADR 0125
agent policies.  The `audit_on_call: true` flag ensures every invocation is
logged with the calling identity.

**Read-vs-write intent** — the tool's MCP annotation `readOnlyHint: false`
and `destructiveHint: false` signal that it can modify state but is not
inherently destructive.  Agents should prefer read-only commands; mutations
should go through governed commands (ADR 0227) when a matching command exists.

### 5. MCP tool surface

The tool registry already supports MCP export via the `export-mcp-tools`
handler.  All 37 tools (including `execute-host-command`) are discoverable as
MCP-compatible tool definitions.

For chat surfaces (LibreChat, Claude Code):
- Claude Code: tools are available via the MCP server at the API gateway
- LibreChat: tools are called via the Actions/OpenAPI integration
- Dify: tools are called via the `/v1/dify-tools/{tool_name}` HTTP endpoint

The MCP schema for `execute-host-command`:
```json
{
  "name": "execute-host-command",
  "description": "Execute a shell command directly on any platform host...",
  "inputSchema": {
    "type": "object",
    "required": ["command"],
    "properties": {
      "command": { "type": "string" },
      "host": { "type": "string", "enum": ["runtime-control-lv3", "proxmox", ...] },
      "timeout": { "type": "integer", "minimum": 1, "maximum": 120 }
    }
  }
}
```

### 6. Audit and traceability

Every `execute-host-command` invocation produces a structured log entry:

```json
{
  "actor_id": "service/dify",
  "tool": "execute-host-command",
  "command": "df -h",
  "host": "runtime-control-lv3",
  "exit_code": 0,
  "output_bytes": 1234,
  "duration_ms": 365,
  "timestamp": "2026-04-09T13:26:50Z"
}
```

This integrates with:
- API gateway structured logging (JSON to stdout, collected by Dozzle/Loki)
- ADR 0354 mutation audit log (when the tool performs write operations)
- ADR 0125 daily execution cap tracking

## Consequences

### Positive
- Agents can autonomously diagnose platform issues (`free -m`, `journalctl -u nginx`,
  `docker ps`, `systemctl status`) without human SSH access
- Multi-host targeting gives agents visibility into the entire platform topology
- MCP compatibility means any MCP-capable chat surface can invoke tools natively
- Safety filter and capability bounds prevent catastrophic operations
- Full audit trail enables post-incident review of every agent action

### Negative
- `Privileged: true` containers are a security surface — mitigated by the safety
  filter and the fact that agents already have Docker socket access
- SSH key provisioning adds a credential to manage — mitigated by Ansible-managed
  lifecycle and the key being on an internal-only network
- nsenter bypasses container isolation — this is intentional; the tool's purpose
  is host-level access, not sandboxed execution

### Risks
- An agent with a compromised identity could use `execute-host-command` to
  exfiltrate data or modify host state — mitigated by ADR 0125 daily caps,
  audit logging, and the safety filter
- SSH key compromise on runtime-control-lv3 grants access to all managed hosts —
  mitigated by internal-only networking (10.10.10.0/24) and key rotation via
  Ansible convergence
