# ADR 0081: Platform Changelog And Deployment History Portal

- Status: Proposed
- Implementation Status: Not Implemented
- Implemented In Repo Version: not yet
- Implemented In Platform Version: not yet
- Implemented On: not yet
- Date: 2026-03-22

## Context

The platform accumulates live-apply receipts in `receipts/live-applies/`. There are now dozens of receipts, each a JSON file recording what was applied, by whom, at what version, and with what verification evidence. But:

- receipts are not browsable in a human-readable form
- there is no timeline view of "what changed on this platform over the last 30 days"
- `changelog.md` is hand-maintained and often incomplete or lags behind actual receipts
- operators cannot quickly answer "when was Grafana last updated?" or "what changed on 2026-02-15?"
- agents synthesising a platform summary must scan dozens of JSON files to reconstruct history

The staging/production promotion model (ADR 0073) will generate promotion receipts in addition to live-apply receipts. The mutation audit log (ADR 0066) records every mutation. All three are complementary but not unified.

What is missing is a generated, human-readable changelog page that synthesises receipts, promotion records, and audit events into a timeline — the deployment history portal.

## Decision

We will generate a **Deployment History Portal** (`changelog.lv3.org`) as part of the operations portal suite (ADR 0074).

### Data sources

The portal is generated from three sources, in order of authority:

1. **Live-apply receipts** (`receipts/live-applies/`): each receipt is a structured JSON record of an Ansible convergence run
2. **Promotion receipts** (`receipts/promotions/`): each record covers a staging-to-production promotion event (ADR 0073)
3. **Mutation audit log** (`/var/log/platform/mutation-audit.jsonl` via Loki, ADR 0052 + ADR 0066): covers out-of-band mutations and command catalog executions

### Portal views

**`/` — Timeline**
Reverse-chronological list of all platform changes. Each entry shows:
- date and time
- change type: `live-apply`, `promotion`, `command-catalog`, `manual`
- actor (operator or agent identity class)
- affected service(s) or VM(s)
- outcome: `success`, `partial`, `failure`
- link to full receipt or audit event

**`/service/<id>` — Per-service history**
Filtered timeline for a single service, showing every deployment, config change, and image upgrade. Links to the ADR and runbook for context.

**`/environment/<env>` — Per-environment history**
All changes in `production` or `staging`, useful for post-incident timeline reconstruction.

**`/promotions` — Promotion log**
Table of all staging-to-production promotions with: branch, promotion date, actor, staging validation duration, any bypasses.

### Generation

The changelog portal is generated by `scripts/generate_changelog_portal.py`:

```
make generate-changelog-portal
```

Inputs:
- `receipts/live-applies/**/*.json`
- `receipts/promotions/**/*.json`
- `config/service-capability-catalog.json` (for service name resolution)
- Loki query for `mutation-audit` log entries (last 90 days, fetched at generation time)

Output:
- `build/changelog-portal/` — static HTML site (same pattern as ops portal)

The generation script runs alongside the ops portal generator in `make generate-status`.

### `changelog.md` replacement

The hand-maintained `changelog.md` in the repository root is replaced by the generated portal. The `Unreleased` working section in `changelog.md` is preserved as a human-authored release notes scratchpad; on version bump it is appended to a `docs/release-notes/<version>.md` file. The detailed history moves to the generated portal.

### Agent query tool

A `get-deployment-history` tool is added to the agent tool registry (ADR 0069):

```json
{
  "name": "get-deployment-history",
  "description": "Return recent platform deployments filtered by service, environment, or date range",
  "category": "observe",
  "input_schema": {
    "service_id": { "type": "string", "required": false },
    "environment": { "enum": ["production", "staging"], "required": false },
    "days": { "type": "integer", "default": 30 }
  }
}
```

This allows agents to answer "what changed in production in the last 7 days?" without scanning raw JSON files.

## Consequences

- Operators have a human-readable timeline of every platform change, replacing a stale hand-maintained changelog.
- Post-incident reviews have a structured timeline to reconstruct what changed before an incident, generated from the same receipts used for verification.
- The Loki dependency for the audit log means the full changelog requires Loki to be running at generation time; if Loki is unavailable, the changelog falls back to receipts-only (structural changes only, no command catalog events).
- `changelog.md` losing its detailed history role requires updating any processes or runbooks that reference it as an authoritative history.

## Boundaries

- The deployment history portal is a read surface. It does not allow re-applying or reverting changes.
- Receipt data older than 90 days in Loki (if log retention is set to 90 days) will not appear in the full audit layer of the timeline; receipt files in `receipts/` are permanent and always available.
- The portal covers platform-level changes only. Application-level changes within a service (e.g. a Windmill workflow definition update) are recorded in the mutation audit log but are not surfaced in the service timeline by default.
