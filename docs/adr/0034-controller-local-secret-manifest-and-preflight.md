# ADR 0034: Controller-Local Secret Manifest And Preflight

- Status: Accepted
- Implementation Status: Partial Implemented
- Implemented In Repo Version: 0.35.0
- Implemented In Platform Version: not applicable (repo-only)
- Implemented On: 2026-03-22
- Date: 2026-03-22

## Context

This repository intentionally keeps sensitive material outside git, mostly under `.local/`.

That is the right security direction, but the codebase still has a maintainability problem:

- secret prerequisites are discovered piecemeal
- different playbooks fail at different stages when a local file is missing
- operators and assistants have to infer which secrets are required for which workflow
- `.local/` is part of the real operating contract, but not yet described in a single machine-readable way

As more services are added, missing-secret surprises will become a larger source of failed runs.

## Decision

We will define a controller-local secret manifest and a standard preflight check.

The manifest will record:

1. Which local secret files or environment variables each workflow requires.
2. Whether a secret is generated by repo automation or provided externally.
3. Which playbooks or scripts depend on each secret.
4. Which secrets are optional, blocked, or required for a given path.
5. A preflight command that checks prerequisites before a long converge starts.

## Consequences

- Operators get one predictable place to check prerequisites.
- Playbooks fail earlier and with better messages.
- Repository onboarding becomes easier because the local operating contract is explicit.
- The manifest itself must stay small and factual; it should describe secret dependencies, not become a second automation system.
