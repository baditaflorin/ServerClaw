# ADR 0074: Platform Operations Portal

- Status: Proposed
- Implementation Status: Not Implemented
- Implemented In Repo Version: not yet
- Implemented In Platform Version: not yet
- Implemented On: not yet
- Date: 2026-03-22

## Context

The platform now runs more than a dozen services across six VMs. Reaching any of them requires knowing: which VM it runs on, which internal IP and port it listens on, which subdomain (if published), which identity class to authenticate with, and which runbook covers its operations. None of this is surfaced in one place.

An operator — or an agent — answering "where does X live?" currently must either:
1. read `versions/stack.yaml`, cross-reference `config/uptime-kuma/monitors.json`, and search the runbooks directory, or
2. recall from memory

As the service count grows, this recall-from-memory model scales poorly. New team members and new agents both face the same cold-start problem.

What is missing is a single, browsable, human-readable dashboard that answers for every service: what it is, where it runs, how to reach it, what its health is, and where to find its runbook. This is the "operations portal" — the authoritative map of the platform for humans navigating it.

## Decision

We will deploy a **Platform Operations Portal** (`ops.lv3.org`) as a lightweight static web application served from the NGINX edge.

### Design principles

- **generated, not hand-maintained** — the portal is rendered from `config/service-capability-catalog.json` (ADR 0075) on every merge to main, not edited by hand
- **read-only, always public to authenticated operators** — the portal displays state; it does not issue commands
- **service-map first** — the primary view is a grid of service cards grouped by VM and function
- **linked to live health** — each service card pulls its current status from Uptime Kuma's API
- **zero custom backend** — the portal is a static HTML/JS build generated by a script; it has no server-side runtime to maintain

### Portal structure

**`/` — Service Map**
Grid of service cards. Each card shows:
- service name and one-line description
- current health indicator (green/yellow/red) from Uptime Kuma
- primary URL (internal or public)
- VM host
- quick links: UI | Runbook | Grafana dashboard

Cards are grouped by function: Observability, Security, Automation, Data, Communication, Access.

**`/vms` — VM Inventory**
Table of all VMs from `stack.yaml`: VMID, hostname, IP, role, running status, services hosted.

**`/subdomains` — DNS Map**
All active subdomains: name, target VM, exposure (public/private), TLS status, certificate expiry date.

**`/runbooks` — Runbook Index**
Searchable index of all runbooks in `docs/runbooks/`. Each entry shows the runbook title, the ADR it maps to, and a direct link to the markdown file in the repo viewer.

**`/adrs` — Decision Log**
Table of all ADRs: number, title, status (proposed/implemented), implementation date. Filterable by status.

**`/agents` — Agent Capability Surface**
Lists all tools in `config/agent-tool-registry.json` (ADR 0069): name, category, description, whether approval is required.

### Generation pipeline

```
make generate-ops-portal
```

Script: `scripts/generate_ops_portal.py`

Inputs:
- `config/service-capability-catalog.json` (ADR 0075)
- `versions/stack.yaml`
- `config/uptime-kuma/monitors.json`
- `config/agent-tool-registry.json`
- `docs/adr/*.md` (front-matter parsed for status/date)
- `docs/runbooks/*.md` (title from H1)

Output:
- `build/ops-portal/` — static HTML site

The generation script runs as part of the `make generate-status` target and during the CI validate pipeline. The resulting `build/ops-portal/` directory is served by the NGINX edge at `ops.lv3.org`.

### Authentication

The portal is served behind NGINX auth via Keycloak (ADR 0056) using the `operator` role. It is not publicly accessible without authentication.

## Consequences

- Operators gain a single entry point for navigating the platform instead of consulting multiple files and runbooks.
- The portal is always in sync with the canonical state files because it is generated on merge; there is no risk of it drifting from reality.
- The generation script must be updated whenever a new catalog field is added; this is a lightweight maintenance cost.
- Serving via NGINX means the portal is only reachable when the nginx-lv3 VM is running; the portal itself does not require any additional VM or container.

## Boundaries

- The portal is a read surface. Initiating workflows, approving commands, or mutating platform state is handled by Windmill (ADR 0044), the command catalog (ADR 0048), and the agent tool registry (ADR 0069) respectively.
- The portal does not replace Grafana dashboards for metrics, Loki for logs, or GlitchTip for error tracking. It links to all of these.
- The staging environment has its own portal instance at `ops.staging.lv3.org` generated from the staging service catalog.
